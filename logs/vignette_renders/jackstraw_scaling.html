The question

After RaJIVE finds a joint axis, the next question is more local: which features are actually tied to it? jackstraw_rajive() answers with a score-permutation approximation. It samples feature rows, permutes the joint score to break the association expected under the null, and compares observed F-statistics with the accumulated null statistics.

The cost has three moving parts:

  • n (samples): more rows increase the linear algebra cost
  • p (features per block): more observed F-statistics are scored
  • n_null (null replicates): the user-controlled accuracy/cost dial

This vignette asks how those three parts behave in practice, then turns the timings into a rule of thumb for choosing n_null.

All timing and memory runs use the current default multiple-testing behavior, correction = "BH" (Benjamini-Hochberg), applied globally across all block/component/feature tests in each run. Use correction = "BY" when you want the more conservative Benjamini-Yekutieli procedure under arbitrary feature dependence.

Time vs Sample Size (n)

ns <- c(50L, 100L, 200L)

time_n_rows <- lapply(ns, function(n_i) {
  set.seed(42)
  Y_i    <- ajive.data.sim(K = 2, rankJ = 2, rankA = c(5, 5),
                           n = n_i, pks = c(500, 500))
  blks_i <- Y_i$sim_data
  fit_i  <- Rajive(blks_i, c(5, 5), joint_rank = 2L,
                   num_cores = 2L, seed = 42L)
  t_i    <- system.time(
    jackstraw_rajive(fit_i, blks_i, n_null = 10L)
  )["elapsed"]
  data.frame(n = n_i, elapsed = t_i)
})

Time vs n_null

n_null is the knob analysts actually turn. Because the permutation loop runs once per null replicate, the elapsed time should rise almost linearly.

profmem: allocation profile on small data

On tiny data (n = 50, p = 100), profmem can show which R call stacks allocate most of the R heap while still running quickly in the vignette.

if (requireNamespace("profmem", quietly = TRUE)) {
  set.seed(1)
  Y_tiny    <- ajive.data.sim(K = 2, rankJ = 2, rankA = c(3, 3),
                               n = 50, pks = c(100, 100))
  blks_tiny <- Y_tiny$sim_data
  fit_tiny  <- Rajive(blks_tiny, c(3, 3), joint_rank = 2L,
                      num_cores = 1L, seed = 42L)

  pm <- profmem::profmem(jackstraw_rajive(fit_tiny, blks_tiny, n_null = 5L))

  # Top 10 allocating calls by total bytes
  pm_df <- as.data.frame(pm)
  pm_df <- pm_df[!is.na(pm_df$bytes), ]
  pm_agg <- aggregate(bytes ~ calls, data = pm_df, FUN = sum)
  pm_top <- pm_agg[order(-pm_agg$bytes)[1:min(10, nrow(pm_agg))], ]
  pm_top$MB <- round(pm_top$bytes / 1024^2, 2)
  knitr::kable(pm_top[, c("calls", "MB")],
               row.names = FALSE,
               col.names = c("Call stack (truncated)", "Allocated (MiB)"),
               caption   = "Top 10 allocating call stacks in jackstraw_rajive() (n=50, p=100, n_null=5)")
} else {
  message("profmem not installed; skipping allocation profile")
}

The practical pattern is simple: start small, learn the signal, then spend the replicates where it matters. Because n_null scales nearly linearly, an exploratory run at n_null = 10 is cheap insurance before increasing to 50-100 for the final analysis.
